## 1. What Custody Means for Agent Work

Custody is the property that every agent action on the Windows PC lane is:

- **Declared** — the action's scope is documented before execution
- **Bounded** — the action cannot exceed its declared scope
- **Observed** — starting and ending state are recorded
- **Receipted** — a verifiable record of the action exists
- **Attributable** — the action is traceable to a specific sprint and operator

This is not about restricting the agent's capability. It is about ensuring that **every capability has a defined boundary and a receipt trail**. An agent with full filesystem access but no custody model is dangerous. An agent with the same access and a custody sandbox is auditable.

## 2. Sandbox Layers

The custody sandbox has three concentric layers:

```
                    ┌─────────────────────────┐
                    │   Layer 3: Authority     │
                    │   (Human Owner)          │
                    │   • Approve sprints      │
                    │   • Approve service      │
                    │     mutation             │
                    │   • Approve model        │
                    │     activation           │
                    └───────────┬─────────────┘
                                │
              ┌─────────────────▼─────────────────┐
              │   Layer 2: Policy                  │
              │   (Sprint Contract)               │
              │                                    │
              │  • Sprint doc defines scope        │
              │  • Sprint receipt proves scope     │
              │  • No action outside scope         │
              │  • No sprint beyond planned        │
              │    sequence without re-plan        │
              └─────────────────┬─────────────────┘
                                │
              ┌─────────────────▼─────────────────┐
              │   Layer 1: Mechanical              │
              │   (Harness Tools)                  │
              │                                    │
              │  • Pre-flight check                │
              │  • Post-flight check               │
              │  • Receipt generation              │
              │  • Baseline health monitoring      │
              │  • Drift detection                 │
              └───────────────────────────────────┘
```

### 2.1 Layer 1 — Mechanical (Harness Tools)
- Automated checks that run without Owner intervention
- Verifies environment state matches expected baseline
- Detects and reports drift but does not auto-correct
- Generates receipts for every action
- **Scope:** Scripts in `scripts/harness/` or equivalent

### 2.2 Layer 2 — Policy (Sprint Contract)
- The sprint doc IS the contract between agent and Owner
- It specifies: files to change, files to not touch, acceptance gates, validation commands
- The agent cannot broaden scope without a new sprint or explicit Owner approval
- Each sprint produces a receipt that proves the contract was fulfilled or explains why not
- **Scope:** Files in `docs/sprints/`, `docs/planning/`, `docs/receipts/`

### 2.3 Layer 3 — Authority (Human Owner)
- The Owner can approve, reject, or defer any sprint
- Service mutation (install, start-type change, binary replacement) requires Owner approval
- Model activation (downloading, selecting a backend for inference) requires Owner approval
- Firewall or network boundary changes require Owner approval
- **Scope:** Owner-controlled gates; no agent can bypass them

## 3. Sandbox Boundaries

### 3.1 Permitted Paths

The sandbox allows writes to:

| Path | Reason |
|------|--------|
| `docs/` | Sprint docs, planning, receipts, operations |
| `scripts/` (new files) | New harness scripts, test files |
| `reports/` | Evidence, measurement outputs |
| `config/` (local only) | `*.local.*` files (gitignored) |
| `logs/` (gitignored) | Runtime logs |
| `temp/` (gitignored) | Temporary work files |
| `fixtures/` | Test fixtures generated by sprints |

### 3.2 Forbidden Targets

The sandbox blocks writes to:

| Path | Reason |
|------|--------|
| `runtime/llama.cpp/` | Model binary location — only changed via explicit backend swap sprint |
| `runtime/bin/` | NSSM binary — only changed via explicit service sprint |
| `models/` | Model files — never committed |
| `.git/` | Git internals — never mutated directly |
| `rust-router/target/` | Build artifacts — gitignored |
| `config/runtime-node.local.json` | Local config — gitignored, may contain secrets |
| `runtime/model_manager.ps1` | Only changed via explicit custody sprint |
| `router/router.py` | Python router — only changed via explicit router sprint |

### 3.3 Service Boundaries

| Action | Sandbox Rule |
|--------|-------------|
| Start `LibrarianRunTimeNode` | Requires Layer 3 (Owner) approval AND admin elevation |
| Stop `LibrarianRunTimeNode` | Requires Layer 3 approval AND admin elevation |
| Change startup type | Requires explicit sprint + Owner approval |
| Install/remove service | Requires explicit sprint + Owner approval |
| Modify NSSM config | Requires explicit sprint + Owner approval |
| Query service state | Always permitted (Layer 1) |

### 3.4 Network Boundaries

| Action | Sandbox Rule |
|--------|-------------|
| Bind router to `127.0.0.1` | Default — always permitted |
| Bind to LAN/WAN `0.0.0.0` | Requires Owner approval + explicit sprint |
| Change firewall rule | Never permitted without Owner approval |
| Open new port | Requires explicit sprint scope |

### 3.5 Process Boundaries

| Action | Sandbox Rule |
|--------|-------------|
| List processes | Always permitted |
| Kill known orphan (`llama-server`, `rust-router` without service) | Permitted if documented in sprint scope |
| Kill unknown process | Never permitted |
| Kill Python router | Only if part of documented lifecycle test |

## 4. Receipt Types

The sandbox defines three receipt types:

### 4.1 Sprint Receipt
Emitted per sprint closeout. Records:
- Sprint ID, starting HEAD, ending HEAD
- Files changed (added, modified)
- Service state before/after
- Port state before/after
- Orphan count before/after
- Acceptance gate results
- Overall status (pass, partial, fail)

### 4.2 Action Receipt
Emitted per discrete action (service start, port check, model select). Records:
- Action type, timestamp
- Starting state snapshot
- Result (success, failure, skipped)
- Evidence (PID, response body, exit code)

### 4.3 Baseline Receipt
Emitted at environment baseline creation/update. Records:
- Machine identity, OS version, hardware
- Tool versions
- Service, port, orphan states
- PATH and key environment variables

## 5. Drift Model

The sandbox does not prevent environment drift. It detects and reports it.

| Drift Type | Detection | Action |
|------------|-----------|--------|
| HEAD changed outside sprint | Pre-flight `git status` | Warn, require confirmation |
| Service state changed | Pre-flight `Get-Service` | Warn, require confirmation |
| Port occupied unexpectedly | Pre-flight `netstat` | Warn, require cleanup |
| Orphan process present | Pre-flight process check | Warn, require cleanup |
| C: drive space below threshold | Pre-flight disk check | Warn, may block model/build sprints |
| Tool version changed | Baseline comparison | Log as informational drift |
| PATH changed | Baseline comparison | Log as informational drift |
